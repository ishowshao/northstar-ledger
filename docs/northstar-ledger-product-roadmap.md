# Northstar Ledger 产品 Roadmap 与技术方向

## 一、虚拟公司与产品

### 1.1 虚拟公司

虚拟公司名为 **Northstar Software**，是一家为自由职业者和小型工作室开发本地优先
生产力软件的小公司。

公司资源有限，因此产品强调：

- 本地即可运行，不依赖复杂云基础设施。
- 用户数据可导入、可导出、可备份。
- 财务计算必须可靠，错误要尽早暴露。
- 功能逐步增长，同时需要长期维护已有数据。
- CLI 与 Web 界面服务于同一套业务能力。

### 1.2 核心产品

产品名为 **Northstar Ledger**。

它是一套面向自由职业者和小型工作室的本地财务与经营台账系统。用户可以记录收入、
支出、客户、项目和发票，从银行或其他工具导入数据，并生成基础经营报表。

选择这个方向的原因是，它能自然产生多种有工程价值的任务：

- 金额精度、时区、日期边界和重复数据等正确性问题。
- CSV 等外部数据格式带来的脏数据和兼容性问题。
- SQLite schema 演进、事务、并发和数据恢复问题。
- Web、CLI、后台任务和文件操作之间的跨模块需求。
- 随数据量增长出现的性能、内存和大输出问题。
- 长期迭代中出现的重构、历史决策和向后兼容取舍。

## 二、产品目标

Northstar Ledger 的长期目标是让一个小型工作室能够在自己的电脑上完成日常经营记录，
并始终保有对数据的控制权。

产品最终应覆盖：

- 客户与项目管理。
- 收入、支出和账户流水管理。
- CSV 导入、校验、预览和去重。
- 发票创建、状态跟踪与导出。
- 月度、季度和年度经营报表。
- 数据备份、恢复与完整性检查。
- 自动化规则和定期任务。
- Web 界面与可脚本化 CLI。
- 可诊断的运行日志与错误报告。

产品不是专业会计或报税软件，也不负责连接真实银行、处理支付或提供合规意见。这些能力
会显著增加外部依赖与风险，不适合作为项目早期范围。

## 三、工程原则

项目在长期开发中遵循以下原则：

1. **正确性优先。** 金额、日期和数据归属不能依赖模糊推断；无法确认时应明确报错。
2. **Fast-fail。** 配置、数据格式或数据库状态不合法时，在靠近问题来源的位置终止，
   不让无效状态继续传播。
3. **本地优先。** 核心功能在无网络环境下可用，SQLite 是主要持久化方式。
4. **可恢复。** 导入、迁移和批处理应考虑失败后的数据一致性与恢复路径。
5. **可测试。** 业务逻辑尽量与 UI、文件系统和进程边界分离，关键规则有自动化测试。
6. **渐进演进。** 每个 roadmap 阶段形成可运行产品，不提前实现尚未需要的复杂抽象。
7. **真实维护。** 允许项目积累有限技术债，再通过后续 issue 处理，而不是一开始追求
   理想化架构。

## 四、总体技术方向

### 4.1 基础语言与运行环境

- 语言：TypeScript，启用严格类型检查。
- Runtime 与包管理：Bun。
- 模块系统：ESM。
- 仓库形态：单仓库，按应用和共享包组织。
- 主要开发平台：macOS 与 Linux。
- 配置方式：环境变量加本地配置文件。

整个项目尽量保持 TypeScript 技术栈统一，使任务重点落在产品工程和系统边界上，而不是
多语言构建环境。

### 4.2 后端与业务层

- HTTP 框架：Hono。
- API 风格：JSON over HTTP，以常规 REST 资源为主。
- 运行时校验：Zod。
- 业务逻辑：独立 TypeScript service/domain 模块，避免直接耦合 HTTP handler。
- 错误处理：结构化业务错误，边界处统一转换为 CLI 或 HTTP 输出。
- API 文档：OpenAPI，在 API 稳定后逐步补充。

项目早期采用单进程本地服务，不引入微服务、消息队列或分布式事务。需要异步执行的工作
先由同一应用中的任务系统承担。

### 4.3 数据存储

- 主数据库：SQLite。
- ORM / 查询构建：Drizzle ORM。
- 数据库迁移：Drizzle Kit 管理的显式迁移文件。
- 金额表示：数据库中存储最小货币单位的整数，领域层使用明确的 Money 类型。
- 时间表示：持久化标准时间和必要的时区信息，展示时再转换。
- 搜索：早期使用 SQLite 查询，后期按需要引入 FTS5。

不在早期引入 PostgreSQL、Redis 或外部搜索服务。SQLite 的锁、事务、迁移、备份和并发
限制本身就是该项目需要真实面对的工程问题。

### 4.4 Web 应用

- UI 框架：React。
- 构建工具：Vite。
- 路由：React Router。
- 服务端状态：TanStack Query。
- 表单：React Hook Form 加 Zod。
- 样式：CSS Modules 与 CSS variables。
- 图表：Recharts，在报表阶段引入。
- 可访问性：优先使用原生语义元素，并通过自动化检查逐步完善。

Web 应用以桌面浏览器为主要目标，第一阶段不做 SSR、移动原生应用或复杂设计系统。

### 4.5 CLI

- CLI 框架：Commander.js。
- 输出：默认适合人阅读，同时为关键命令提供 JSON 输出。
- 交互：基础确认和提示可使用轻量 prompt 库。
- 长任务：显示明确进度，并支持非交互环境下的稳定日志输出。

CLI 不只是开发辅助工具。它是正式产品界面之一，后续将承担导入、导出、备份、检查和
自动化任务，也用于验证相同业务能力是否能在不同 presentation 层复用。

### 4.6 后台任务

- 运行方式：应用内任务队列与 worker，不依赖外部队列服务。
- 初期任务：CSV 导入、报表生成、备份和完整性检查。
- 状态：queued、running、completed、failed、cancelled。
- 观测：结构化日志、进度和最终结果摘要。

后台任务的持久化、重试、取消和进程重启恢复不要求一次完成，而是在 roadmap 中逐步
增强。这一领域应成为持续产生并发与生命周期 issue 的主要来源。

### 4.7 测试与质量工具

- 单元测试和集成测试：`bun test`。
- Web 组件测试：Testing Library。
- 端到端测试：Playwright。
- API 测试：直接调用 Hono 应用或本地 HTTP 服务。
- 数据库测试：每个测试使用独立临时 SQLite 数据库。
- 类型检查：TypeScript `tsc --noEmit`。
- Lint：ESLint。
- 格式化：Biome。
- 覆盖率：用于发现空白区域，不设置鼓励低价值测试的全局硬指标。

### 4.8 日志与诊断

- 日志格式：结构化 JSON 日志与适合本地开发的人类可读输出。
- 日志库：Pino。
- 关联信息：request、task、import 等操作使用明确 ID。
- 错误报告：保存错误类别、上下文摘要和 cause chain，不记录不必要的财务正文。
- 指标：早期只记录进程内或本地指标，暂不接入云监控平台。

### 4.9 交付与运行

- 本地开发：Bun scripts 启动 API、Web 和 worker。
- 打包：先提供源码运行方式，稳定后再提供单机发行包。
- CI：GitHub Actions，执行格式、类型、lint、测试和构建检查。
- 容器：中期提供 Dockerfile，主要用于可重复集成测试和自托管体验。
- 版本管理：语义化版本，数据库 schema 版本独立记录。

## 五、产品 Roadmap

Roadmap 不规定具体 issue 数量。每个阶段应拆成若干独立 issue，并混合功能开发、bug、
测试补充、文档和重构任务。阶段之间允许穿插上一阶段的维护问题。

### R0：仓库与工程基线

目标是建立一个能够运行、测试和持续演进的空产品骨架。

主要成果：

- 建立单仓库和基础 workspace。
- 配置 TypeScript、Biome、ESLint 和测试框架。
- 建立 API、Web、CLI 和共享模块的最小入口。
- 建立 SQLite 连接和第一条迁移。
- 增加统一的开发、检查和测试命令。
- 配置基础 CI。
- 编写贡献说明和本地启动文档。

这一阶段只证明各入口可运行，不提前实现完整业务框架。

### R1：可用的本地台账 MVP

目标是完成第一条端到端业务链路，让用户可以真正记录和查看收支。

主要成果：

- 账户、客户、项目和交易的基本管理。
- 收入与支出的新增、修改、删除和查询。
- 金额、币种、日期和必填字段校验。
- Web 中的交易列表、筛选和编辑表单。
- CLI 中的交易新增、查询和汇总命令。
- 基础月度收入、支出和结余汇总。
- 核心业务规则与 API 集成测试。

阶段结束时，产品应当能供一个虚拟工作室进行日常手工记账。

### R2：真实数据导入与数据质量

目标是接受来自外部系统的不完美数据，并保证导入过程可解释、可回滚。

主要成果：

- CSV 文件解析、字段映射和导入预览。
- 多种日期、金额和字符编码格式处理。
- 重复记录检测与冲突报告。
- 原子导入与失败回滚。
- 导入历史、错误明细和结果摘要。
- 大文件导入的后台执行和进度展示。
- 导入后的数据完整性检查。

### R3：发票与经营报表

目标是让台账从记录工具扩展为基础经营管理工具。

主要成果：

- 发票草稿、编号、发送状态和付款状态。
- 发票条目、税费、折扣和金额汇总。
- 发票导出为 HTML 或 PDF。
- 项目盈利、客户收入和现金流报表。
- 按月份、季度和自定义周期筛选。
- Web 图表与 CLI 报表输出。
- 报表生成任务和结果缓存。

### R4：自动化、任务恢复与可靠性

目标是让产品能够承担重复工作，并在异常退出后保持可诊断、可恢复。

主要成果：

- 定期交易和简单分类规则。
- 计划执行的导入、备份和报表任务。
- 后台任务取消、重试和超时。
- 应用重启后的任务状态恢复。
- 数据库备份、恢复和完整性验证。
- 数据库迁移失败的安全处理。
- 并发启动和重复任务防护。

### R5：规模化与多工作区

目标是验证产品在更大数据量和更复杂使用方式下是否仍然可靠。

主要成果：

- 多工作区及严格的数据隔离。
- 更完整的角色和只读访问能力，但仍以本地部署为主。
- 数十万交易下的查询和报表性能优化。
- 分页、流式导出和大结果限制。
- SQLite 索引、查询计划和 FTS5 搜索优化。
- 长时间运行下的内存与资源使用治理。
- 数据归档和按需恢复。

### R6：发布质量与长期维护

目标是把已经多次演进的项目整理成可交付、可升级的软件。

主要成果：

- 稳定的版本升级与 schema 迁移流程。
- Docker 自托管运行方式和本地发行包。
- 安装、备份、恢复和故障排查文档。
- API 和 CLI 的兼容性政策。
- 安全检查、依赖升级和敏感日志治理。
- 无障碍、跨平台和浏览器兼容性修复。
- 对早期技术债进行有依据的重构。


## 六、阶段验收方式

每个 roadmap 阶段完成后，从以下几个层面验收：

- **产品验收：** 阶段目标是否形成用户可操作的完整链路。
- **正确性验收：** 公开测试和仓库外隐藏测试是否通过。
- **数据验收：** 失败、取消和升级后是否保持一致且可恢复。
- **工程验收：** 类型、格式、lint、测试和构建是否全部通过。
- **运行验收：** CLI、Web、API 和后台任务是否做过真实 smoke test。
- **维护验收：** 文档是否反映当前行为，是否留下无意的双实现或废弃路径。
